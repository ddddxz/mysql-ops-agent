MySQL 高可用架构设计指南

一、高可用架构类型

1. 主从复制架构
   - 一主多从，读写分离
   - 故障时需手动或脚本切换
   - RTO：分钟级
   - 适用场景：读多写少，允许短暂不可用

2. MHA (Master High Availability)
   - 自动故障检测和主从切换
   - 基于 SSH 的管理节点
   - 切换时间：10-30 秒
   - 适用场景：中小规模，要求自动切换

3. MGR (MySQL Group Replication)
   - MySQL 官方高可用方案
   - 基于 Paxos 协议
   - 支持单主和多主模式
   - 切换时间：秒级
   - 适用场景：MySQL 5.7+，要求强一致性

4. MySQL InnoDB Cluster
   - MySQL 官方完整解决方案
   - MGR + MySQL Router + MySQL Shell
   - 自动路由和故障转移
   - 适用场景：新项目，完整高可用方案

5. 云厂商高可用
   - AWS RDS Multi-AZ
   - 阿里云 RDS 高可用版
   - 腾讯云 MySQL 高可用版
   - 切换时间：秒级到分钟级

二、MHA 部署配置

1. 架构要求
   - 至少 3 台服务器：1 主 2 从
   - 管理节点可部署在从库或独立服务器
   - SSH 免密登录

2. 配置文件 (masterha_default.cnf)
   [server default]
   user = mha
   password = mha_password
   ssh_user = root
   repl_user = repl
   repl_password = repl_password
   manager_workdir = /var/log/mha
   manager_log = /var/log/mha/manager.log
   remote_workdir = /var/log/mha
   ping_interval = 1
   secondary_check_script = masterha_secondary_check
   master_ip_failover_script = /usr/local/bin/master_ip_failover
   shutdown_script = ""
   report_script = /usr/local/bin/send_report

   [server1]
   hostname = 192.168.1.1
   candidate_master = 1

   [server2]
   hostname = 192.168.1.2
   candidate_master = 1

   [server3]
   hostname = 192.168.1.3
   no_master = 1

3. 常用命令
   # 检查 SSH 连接
   masterha_check_ssh --conf=/etc/mha.cnf
   
   # 检查复制状态
   masterha_check_repl --conf=/etc/mha.cnf
   
   # 启动管理器
   masterha_manager --conf=/etc/mha.cnf
   
   # 检查状态
   masterha_check_status --conf=/etc/mha.cnf

三、MGR 部署配置

1. 单主模式配置
   [mysqld]
   server_id = 1
   gtid_mode = ON
   enforce_gtid_consistency = ON
   binlog_checksum = NONE
   log_bin = binlog
   log_slave_updates = ON
   binlog_format = ROW
   master_info_repository = TABLE
   relay_log_info_repository = TABLE
   transaction_write_set_extraction = XXHASH64
   loose-group_replication_group_name = "aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa"
   loose-group_replication_start_on_boot = OFF
   loose-group_replication_local_address = "192.168.1.1:33061"
   loose-group_replication_group_seeds = "192.168.1.1:33061,192.168.1.2:33061,192.168.1.3:33061"
   loose-group_replication_bootstrap_group = OFF
   loose-group_replication_single_primary_mode = ON

2. 启动 MGR
   # 首节点引导
   SET GLOBAL group_replication_bootstrap_group=ON;
   START GROUP_REPLICATION;
   SET GLOBAL group_replication_bootstrap_group=OFF;
   
   # 其他节点加入
   START GROUP_REPLICATION;

3. 查看状态
   SELECT * FROM performance_schema.replication_group_members;
   SELECT * FROM performance_schema.replication_group_member_stats;

四、故障切换流程

1. 自动切换条件
   - 主库不可达（网络/进程崩溃）
   - 从库数据同步延迟在允许范围内
   - 有候选主库可用

2. 切换步骤
   - 检测主库故障
   - 选择最新从库作为新主库
   - 应用差异 relay log
   - 提升新主库
   - 其他从库指向新主库
   - 更新 VIP 或 DNS

3. 手动切换
   # MHA 手动切换
   masterha_master_switch --conf=/etc/mha.cnf --master_state=alive --new_master_host=192.168.1.2
   
   # MGR 切换主库
   SELECT group_replication_set_as_primary('member_uuid');

五、监控告警

1. 关键监控指标
   - 主库存活状态
   - 复制延迟
   - 从库状态
   - MGR 成员状态
   - VIP 绑定状态

2. 告警规则
   - 主库宕机：立即告警
   - 复制延迟 > 60s：警告
   - 从库宕机：警告
   - MGR 成员离线：警告

六、Agent 诊断规则

当用户询问高可用问题时，Agent 应：

1. 检查当前架构
   SHOW VARIABLES LIKE 'server_id';
   SELECT * FROM performance_schema.replication_group_members;

2. 检查复制状态
   SHOW SLAVE STATUS;
   SHOW MASTER STATUS;

3. 根据架构类型给出建议：
   - 主从架构：建议升级到 MHA 或 MGR
   - MHA：检查配置和切换脚本
   - MGR：检查成员状态和网络

4. 提供架构选型建议：
   - 小规模：MHA
   - 新项目：MGR/InnoDB Cluster
   - 云环境：使用云厂商方案

七、最佳实践

1. 至少 3 节点部署，避免脑裂
2. 配置自动故障切换
3. 定期进行故障演练
4. 监控复制延迟
5. 使用 VIP 或代理层对外服务
6. 文档化切换流程
7. 设置合理的告警阈值
