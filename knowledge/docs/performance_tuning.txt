MySQL 性能调优综合指南

一、服务器参数调优

1. 内存相关参数

innodb_buffer_pool_size
- 用途：InnoDB 缓冲池，缓存数据和索引
- 建议：系统内存的 70-80%
- 公式：innodb_buffer_pool_size = 总内存 * 0.75
- 监控：SHOW STATUS LIKE 'Innodb_buffer_pool%';

innodb_buffer_pool_instances
- 用途：减少缓冲池争用
- 建议：buffer_pool_size > 1GB 时设置为 4-8

key_buffer_size
- 用途：MyISAM 索引缓存
- 建议：如果不用 MyISAM，设置为 32M 即可

query_cache_type / query_cache_size
- MySQL 8.0 已移除
- MySQL 5.7 建议关闭，高并发下性能下降

2. I/O 相关参数

innodb_io_capacity
- 用途：InnoDB 每秒 I/O 操作数
- 建议：SSD 设置 2000-5000，HDD 设置 200-500

innodb_io_capacity_max
- 用途：I/O 容量上限
- 建议：设置为 innodb_io_capacity 的 2 倍

innodb_flush_method
- 用途：数据刷新方式
- 建议：Linux 使用 O_DIRECT

innodb_file_per_table
- 用途：每表独立表空间
- 建议：ON，便于空间回收

3. 日志相关参数

innodb_log_file_size
- 用途：redo log 文件大小
- 建议：buffer pool 的 25%
- 计算：观察 Innodb_os_log_written 增长速度

innodb_log_buffer_size
- 用途：redo log 缓冲区
- 建议：16M-64M

innodb_flush_log_at_trx_commit
- 1：最安全，每次提交刷新（默认）
- 2：每秒刷新，可能丢失 1 秒数据
- 0：操作系统控制，性能最高

sync_binlog
- 0：由操作系统控制
- 1：每次提交同步（默认，最安全）
- N：每 N 次提交同步

4. 连接相关参数

max_connections
- 用途：最大连接数
- 建议：根据并发需求设置，默认 151
- 计算：每个连接约 256KB-1MB 内存

thread_cache_size
- 用途：线程缓存
- 建议：设置为 max_connections 的 10-20%

wait_timeout / interactive_timeout
- 用途：连接超时时间
- 建议：8 小时（28800 秒）或更短

二、查询优化

1. 索引优化原则

- 为 WHERE、JOIN、ORDER BY 列创建索引
- 使用复合索引优化多列查询
- 遵循最左前缀原则
- 避免过度索引，影响写入性能
- 定期分析索引使用情况

查看索引使用情况：
SELECT * FROM sys.schema_unused_indexes;
SELECT * FROM sys.schema_redundant_indexes;

2. 查询改写技巧

- 避免 SELECT *，只查询需要的列
- 使用 LIMIT 限制结果集
- 避免在 WHERE 中使用函数
- 使用 EXISTS 替代 IN
- 大分页使用延迟关联

大分页优化示例：
-- 原始
SELECT * FROM orders LIMIT 100000, 20;
-- 优化
SELECT o.* FROM orders o 
INNER JOIN (SELECT id FROM orders LIMIT 100000, 20) t 
ON o.id = t.id;

3. JOIN 优化

- 小表驱动大表
- 确保 JOIN 列有索引
- 避免 JOIN 过多表（建议不超过 5 个）
- 使用 EXPLAIN 分析执行计划

三、表结构优化

1. 数据类型选择

整数类型：
- TINYINT: 1 字节，范围 -128~127
- SMALLINT: 2 字节
- INT: 4 字节
- BIGINT: 8 字节

字符串类型：
- CHAR: 定长，适合固定长度
- VARCHAR: 变长，适合可变长度
- TEXT: 大文本，避免使用

时间类型：
- DATETIME: 8 字节，范围 1000-9999
- TIMESTAMP: 4 字节，范围 1970-2038
- 建议使用 DATETIME

2. 范式与反范式

范式化优点：
- 减少数据冗余
- 更新操作简单

反范式化优点：
- 减少 JOIN 操作
- 提高查询性能

建议：适度反范式化，空间换时间

四、监控指标

1. 关键性能指标

QPS (Queries Per Second)
SHOW STATUS LIKE 'Questions';

TPS (Transactions Per Second)
SHOW STATUS LIKE 'Com_commit';
SHOW STATUS LIKE 'Com_rollback';

连接数
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';

缓冲池命中率
SHOW STATUS LIKE 'Innodb_buffer_pool_read%';
命中率 = 1 - (Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests)

2. 慢查询监控

-- 启用慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 2;

-- 查看慢查询数量
SHOW STATUS LIKE 'Slow_queries';

-- 分析慢查询
mysqldumpslow -s t /var/log/mysql/slow.log

五、Agent 诊断规则

当用户询问性能问题时，Agent 应：

1. 收集系统信息
   SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
   SHOW VARIABLES LIKE 'max_connections';
   SHOW STATUS LIKE 'Threads%';
   SHOW STATUS LIKE 'Innodb_buffer_pool%';

2. 分析当前负载
   SHOW PROCESSLIST;
   SHOW STATUS LIKE 'Questions';
   SHOW STATUS LIKE 'Slow_queries';

3. 根据指标给出建议：
   - 缓冲池命中率 < 99%：增加 buffer_pool_size
   - 连接数接近 max_connections：增加连接数限制
   - 存在大量慢查询：分析慢查询日志

4. 提供优化 SQL 示例
5. 建议监控告警配置

六、最佳实践

1. 定期进行性能基线测试
2. 监控关键指标并设置告警
3. 定期分析慢查询日志
4. 使用 pt-index-usage 分析索引使用
5. 使用 pt-query-digest 分析查询模式
6. 定期优化表碎片
7. 保持统计信息更新
